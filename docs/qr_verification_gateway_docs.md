## ১. ওভারভিউ ও উদ্দেশ্য {#overview}

**QR Verification Gateway** হলো RoktoDut-এর একটি ট্রাস্ট-লেয়ার সিস্টেম, যা NID-Verified ডোনারদের একটি **ডিজিটাল আইডেন্টিটি** প্রদান করে। এই সিস্টেমের মাধ্যমে:

- হাসপাতাল বা রোগীর স্বজনরা **কিউআর কোড স্ক্যান করে** ডোনারের পরিচয় তাৎক্ষণিকভাবে যাচাই করতে পারেন
- ডোনারকে লগইন বা কথা বলার আগেই বিশ্বাসযোগ্যতা প্রমাণ করতে হয় না
- সিস্টেম নিশ্চিত করে যে শুধুমাত্র প্রয়োজনীয় তথ্যই প্রকাশ পাচ্ছে — না কম, না বেশি

> [!IMPORTANT]
> এই ফিচারটি শুধুমাত্র **NID-Verified** (`nid_status = 'verified'` এবং `verified_badge = true`) ডোনারদের জন্য সক্রিয় থাকে। অপ্রমাণিত অ্যাকাউন্টের জন্য কোনো QR কার্ড জেনারেট হবে না।

## ৩. পাবলিক স্ক্যান: কেন শুধু নাম ও ব্লাড গ্রুপ — Frictionless Trust {#public-scan}

### ৩.১ মূল দর্শন: "Just Enough Trust"

ইমার্জেন্সিতে একজন রোগীর স্বজনের কাছে **দুটি তথ্যই যথেষ্ট**:
1. **এই মানুষটি কে?** → নাম
2. **এই মানুষটির রক্ত আমার রোগীর কাজে লাগবে?** → ব্লাড গ্রুপ

অতিরিক্ত তথ্য প্রদর্শন করলে দুটি সমস্যা তৈরি হয়:
- **জ্ঞানীয় চাপ বৃদ্ধি (Cognitive Overload):** স্ট্রেসের মুহূর্তে বেশি তথ্য সিদ্ধান্ত গ্রহণকে ধীর করে
- **প্রাইভেসি ঝুঁকি:** অপ্রাসঙ্গিক তথ্য প্রকাশ ডোনারের সম্মতি ছাড়াই হয়ে যেতে পারে

### ৩.২ Frictionless Trust মডেল

```
স্ক্যান → ১ সেকেন্ডে সিদ্ধান্ত → যোগাযোগের উদ্যোগ
```

রোগীর স্বজন QR স্ক্যান করার পর যা দেখেন:

```
┌────────────────────────────────┐
│  ✅ রক্তদূত — যাচাইকৃত ডোনার  │
│                                │
│  নাম:       রাহেলা বেগম        │
│  ব্লাড গ্রুপ: B+               │
│  স্ট্যাটাস:  ✅ NID Verified   │
│                                │
│  [📞 যোগাযোগ করুন]             │
└────────────────────────────────┘
```

> [!NOTE]
> "যোগাযোগ করুন" বাটনে ক্লিক করলে ইউজারকে **লগইন পেজে** নিয়ে যাওয়া হয় বা **Math Challenge** সমাধান করতে বলা হয় — সরাসরি ফোন নম্বর একসাথে দেখানো হয় না।

### ৩.৩ বিশ্বস্ততার প্রমাণ কেন যথেষ্ট?

RoktoDut-এর ভেরিফিকেশন প্রক্রিয়ায় একজন ডোনার সম্পর্কে আগে থেকেই নিশ্চিত হওয়া গেছে:
- ✅ তার জাতীয় পরিচয়পত্র (NID) বৈধ
- ✅ তিনি একজন সক্রিয় রক্তদাতা
- ✅ তার পরিচয় সিস্টেম অ্যাডমিন কর্তৃক অনুমোদিত

সুতরাং "তিনি কে?" প্রশ্নটির উত্তর দেওয়া হয়েছে। বাকি তথ্য (ফোন) যোগাযোগের ধাপে আসে, পরিচয় যাচাইয়ের ধাপে নয়।

## ৪. ফোন নম্বর গোপন রাখার কারণ — Privacy Protection {#privacy}

### ৪.১ হুমকির মডেল (Threat Model)

| হুমকি | বিবরণ |
|:---|:---|
| **ডেটা স্ক্র্যাপিং** | বট স্বয়ংক্রিয়ভাবে হাজার হাজার QR লিংক স্ক্যান করে ফোন নম্বর সংগ্রহ করতে পারে |
| **স্প্যামিং / হ্যারাসমেন্ট** | প্রকাশিত নম্বরে অপ্রাসঙ্গিক কল বা হুমকির ঝুঁকি |
| **পরিচয় চুরি (Identity Theft)** | নাম + ফোন + রক্তগ্রুপ একসাথে পেলে সোশ্যাল ইঞ্জিনিয়ারিং সহজ হয় |
| **কমার্শিয়াল অপব্যবহার** | ফোন ডেটা হেলথকেয়ার মার্কেটিং কোম্পানিতে বিক্রির ঝুঁকি |

### ৪.২ Privacy-First ডিজাইন সিদ্ধান্ত

RoktoDut-এর `User` মডেলে `phone` ফিল্ডটি `$hidden` অ্যারেতে আছে:

```php
protected $hidden = [
    'password',
    'remember_token',
    'nid_image',
    'qr_token',   // 🔐 API-তে কখনো expose হবে না
    'phone',      // 🔐 Privacy Shield — serialize-এ দেখা যাবে না
    'email',      // 🔐 Privacy Shield
    'nid_number', // 🔐 Privacy Shield
];
```

এই ডিজাইনের ফলে:
- Laravel-এর যেকোনো `toArray()` বা `toJson()` কলে ফোন নম্বর **স্বয়ংক্রিয়ভাবে বাদ** পড়বে
- API response বা Blade টেম্পলেটে ভুলবশত expose হওয়ার সম্ভাবনা শূন্য

### ৪.৩ ফোন নম্বর দেখার প্রক্রিয়া (Controlled Reveal)

ফোন নম্বর দেখতে হলে ব্যবহারকারীকে পার করতে হবে:

```
১. Math Challenge সমাধান  →  ২. Rate Limit চেক  →  ৩. Revealed Number
                                (১৫ মিনিটে ৫ বার)
```

এই প্রক্রিয়া `PhoneRevealLog` মডেলে লগ করা হয় — ফলে যে কেউ অযৌক্তিকভাবে বারবার নম্বর দেখার চেষ্টা করলে সিস্টেম তা শনাক্ত করতে পারে।

> [!TIP]
> QR স্ক্যান পেজে শুধু "যোগাযোগ করুন" বাটন রাখা হয়েছে। এই বাটনে ক্লিক করলে ব্যবহারকারীকে স্বাভাবিক `PhoneReveal` ফ্লোতে নিয়ে যাওয়া হয়, যেখানে Human Verification (Math Challenge) বাধ্যতামূলক।

## ৫. qr_token সিস্টেম: অ্যাবিউজ প্রতিরোধ কৌশল {#qr-token}

### ৫.১ qr_token কী?

`qr_token` হলো প্রতিটি ডোনারের জন্য একটি **একক, ক্রিপ্টোগ্রাফিক্যালি র‍্যান্ডম** স্ট্রিং, যা:
- ডোনারের ব্যক্তিগত ID (যেমন: `user_id = 42`) **বাইরে প্রকাশ করে না**
- URL-এ ব্যবহৃত হয়: `/verify/abc123xyz...`
- ডেটাবেজে একমাত্র সংযোগসূত্র হিসেবে কাজ করে

```php
// User.php এ সংজ্ঞায়িত
'qr_token' => 'unique, random, 64-char hex string'

// $hidden অ্যারেতে থাকায় API/JSON-এ কখনো দেখা যায় না
'qr_token' // 🔐 JSON/API response-এ কখনো expose করা যাবে না
```

### ৫.২ qr_token কেন Sequential ID-র চেয়ে নিরাপদ?

| দুর্বলতা | Sequential ID (`/verify/42`) | qr_token (`/verify/a3f9...`) |
|:---|:---:|:---:|
| **Enumeration Attack** | ✗ সম্ভব (1, 2, 3...) | ✅ অসম্ভব |
| **User Count অনুমান** | ✗ সম্ভব | ✅ অসম্ভব |
| **Brute Force** | ✗ সহজ | ✅ ২⁵¹² সম্ভাবনা |
| **পরিচয় সংযোগ** | ✗ সরাসরি | ✅ শুধু DB-তে |

### ৫.৩ অ্যাবিউজ প্রতিরোধের স্তরসমূহ

#### স্তর ১ — টোকেন অপ্রকাশযোগ্যতা (Token Opacity)

```
QR Code URL:  /verify/a3f9b2c1d4e5f6...  ← কোনো ব্যক্তিগত তথ্য নেই
     ↓
DB Lookup:    users WHERE qr_token = 'a3f9...' → User #42
     ↓
Response:     { name: "রাহেলা", blood_group: "B+" }  ← সীমিত তথ্য
```

#### স্তর ২ — রেট লিমিটিং (Rate Limiting)

```
/verify/{token}  →  max 30 requests/minute per IP
                     (Laravel Throttle Middleware)
```

অতিরিক্ত স্ক্যান হলে `429 Too Many Requests` রিটার্ন হয় এবং সন্দেহজনক IP লগ করা হয়।

#### স্তর ৩ — টোকেন রিসেট মেকানিজম (Token Rotation)

নিম্নলিখিত ইভেন্টে `qr_token` স্বয়ংক্রিয়ভাবে পরিবর্তিত হয়:
- ডোনার স্বেচ্ছায় **"আমার QR রিফ্রেশ করুন"** চাপলে
- সিস্টেম অ্যাডমিন অ্যাকাউন্ট রিভিউ করলে
- পাসওয়ার্ড পরিবর্তন বা NID পুনরায় ভেরিফাই করলে

পুরনো টোকেন তাৎক্ষণিকভাবে অকার্যকর হয়ে যায় — ফলে শেয়ার করা পুরনো QR কোড আর কাজ করবে না।

#### স্তর ৪ — স্ক্যান লগিং (Audit Trail)

```sql
-- qr_scan_logs টেবিল (প্রস্তাবিত)
id | qr_token_hash | scanner_ip | user_agent | scanned_at | result
```

- স্ক্যানের লগ রাখা হয় কিন্তু `qr_token` সরাসরি নয়, তার **SHA-256 হ্যাশ** সংরক্ষিত হয়
- অস্বাভাবিক স্ক্যান প্যাটার্ন শনাক্ত হলে অ্যাডমিন অ্যালার্ট পাঠানো হয়

#### স্তর ৫ — Shadow-ban Integration

```php
// User.php
public function scopeNotShadowbanned($query)
{
    return $query->where('is_shadowbanned', false);
}
```

কোনো অ্যাকাউন্ট সন্দেহজনক কার্যকলাপে জড়িত হলে `is_shadowbanned = true` করলে তার QR পেজ ভিজিটর ভ্যালিড রেসপন্স পাবে না।

## ৭. সিকিউরিটি মডেল সারসংক্ষেপ {#security-summary}

```
Defense-in-Depth স্তরসমূহ:

Layer 1: qr_token → Sequential ID নয়, enumerate করা অসম্ভব
Layer 2: $hidden → API/JSON serialize-এ sensitive data বাদ
Layer 3: Minimal Disclosure → শুধু নাম + ব্লাড গ্রুপ পাবলিক
Layer 4: Rate Limiting → ৩০ req/min per IP
Layer 5: Math Challenge → ফোন নম্বর reveal-এ human check
Layer 6: Token Rotation → পুরনো QR অকার্যকর করার সুবিধা
Layer 7: Audit Log → সকল স্ক্যান ট্র্যাকড
Layer 8: Shadow-ban → সন্দেহজনক অ্যাকাউন্ট নীরবে নিষ্ক্রিয়
```
